# Unscramble Game - Лабораторная работа №13-14

## Описание

Игра-головоломка, в которой игроку предлагается угадать исходное слово, глядя на перемешанный набор букв.

## Технологии

- Kotlin
- Jetpack Compose
- ViewModel, StateFlow,
- Kotlin Coroutines

## Функциональность

- Отображение текущего перемешанного слова. 
- Поле для ввода ответа пользователем. 
- Кнопка проверки введенного слова. 
- Кнопка перехода к следующему слову (скип). 
- Отображение счета (количество угаданных слов). 
- Обработка состояния игры (правильно/неправильно, конец игры). 
- Сохранение состояния игры при повороте экрана (через ViewModel).

## Контрольные вопросы:

### 1. Что такое ViewModel и зачем он нужен?

- ViewModel разделяет логику работы приложения от его UI, что позволяет сохранять данные при изменение конфигурации и легко добавлять новый функционал, не затрагивая при этом UI
- Т.к. изменяется конфигурация, данные теряются

### 2. В чём разнциа между MutableStateFlow и StateFlow?

- MutableStateFlow (изменяемый) позволяет менять свое значение, а StateFlow (неизменяемый) доступен только для чтения.
- Зачем два поля (_uiState и uiState): Это реализация паттерна "Backing Property".
Мы создаем приватное изменяемое поле _uiState для внутреннего изменения данных во ViewModel, а наружу (в UI) предоставляем его как неизменяемый StateFlow (uiState), чтобы гарантировать, что только ViewModel может обновлять состояние.

### 3. Почему в ViewModel нельзя хранить Context или View

- Риск утечки памяти: ViewModel живет дольше, чем Activity. Если ViewModel хранит прямую ссылку на Context (особенно на Activity или View), то сборщик мусора не сможет уничтожить Activity после её закрытия или пересоздания. Это приводит к утечке памяти и, в конечном итоге, к падению приложения.

### 4. Чем StateFlow отличается от remember и rememberSaveable?

- remember: Хранит состояние только в памяти во время текущей композиции. При повороте экрана состояние сбрасывается.
rememberSaveable: Сохраняет состояние в Bundle. Переживает поворот экрана, но не переживает сворачивание приложения в фоне и его закрытие системой.
StateFlow: Живет вне Compose. Автоматически сохраняет состояние при повороте, но требует дополнительных инструментов для сохранения состояния при Process Death.
- Когда что использовать: remember — для UI-элементов внутри одной композиции.
rememberSaveable — для UI-элементов, состояние которых нужно сохранить при пересоздании UI.
StateFlow — для управления данными игры, счета, списков, т.е. всей логикой приложения.

### 5. Что такое data class в Kotlin и зачем он используется для UI State?

- Data Class - это класс, предназначенный для хранения данных. Идеально подходит для описания состояния экрана.
Мы создаем один неизменяемый класс, который содержит все возможные состояния UI. Это упрощает отладку и тестирование.
- Автоматические методы: toString(), equals(), hashCode(), copy().
- Зачем copy(): Так как data class неизменяем, мы не можем менять его поля.
Функция copy() позволяет создать новую копию объекта с измененными несколькими полями, не трогая старый объект.

### 6. Как работает collectAsState() в Compose?

- collectAsState() - это функция, которая подписывается на StateFlow. Каждый раз, когда в StateFlow появляется новое значение, эта функция получает его и конвертирует в обычный State, знакомый Compose.
- Автоматическое обновление: Когда значение внутри этого нового State меняется, Compose автоматически перекомпоновывает все компоненты, которые читают это состояние.

### 7. Зачем разделять код на пакеты data и ui?

- Разделение на слои поддерживает принципы чистой архитектуры (Clean Architecture):
data (слой данных): Отвечает за получение данных. Не знает ничего об UI.
ui (слой представления): Отвечает за отображение данных на экране и обработку действий пользователя. Взаимодействует с data слоем через ViewModel.
Код становится слабосвязанным, его легче тестировать, масштабировать и модифицировать.

### 8. Что происходит с ViewModel, когда приложение полностью закрывается
(process death)?

- Сохранение данных: По умолчанию, когда ОС убивает процесс приложения в фоне, ViewModel уничтожается, и все данные в ней теряются.
Ссылка на ViewModel, хранящаяся в Activity, также исчезает.
- Альтернативы для долговременного хранения:
SavedStateHandle: Встроенный механизм ViewModel, позволяющий сохранять небольшие объемы данных в Bundle. Идеален для хранения ID экрана или временного текста.
DataStore / SharedPreferences: Для хранения пользовательских настроек.
База данных (Room): Для хранения больших структурированных данных, списков задач, истории игры.
Кэш в памяти: Если данные не критичны и их можно легко перезапросить с сервера.
